# Cluster Verification System

## Overview

The CAMO PKO testing scripts include comprehensive cluster verification to prevent accidental execution on the wrong cluster. This is critical because:

1. **Multi-cluster context switches** - Users often switch between test clusters and Hive management clusters
2. **Script pauses** - Long-running scripts have pauses for manual steps (e.g., Hive sync operations)
3. **Destructive operations** - Some phases delete resources and should only run on the intended test cluster

## How It Works

### 1. Cluster Information Storage

The config file (`.camo-pko-test-config`) stores multiple cluster identifiers:

```bash
# Cluster Verification Data
CLUSTER_ID=YOUR_CLUSTER_NAME              # User-provided cluster name/ID
CLUSTER_UUID=abc123-def456-...          # Unique cluster UUID from clusterversion
CLUSTER_SERVER=https://api....          # API server URL (unique per cluster)
CLUSTER_USER=backplane-cluster-admin     # Username for reference
```

### 2. Verification Points

Cluster verification happens at:

- **Phase 1** - Optional early setup (if already logged into test cluster)
- **Phase 3** - Mandatory verification and cluster ID entry
- **Phase 3 (after Hive pause)** - Critical re-verification after manual Hive operations
- **Phases 4, 5, 6, 7, 8** - Start of each phase
- **Phase 8 (after Hive restore)** - Re-verification after Hive sync restoration

### 3. Multi-Level Verification

The verification function (`verify_cluster()`) performs three checks:

#### Primary Check: Cluster UUID
```bash
oc get clusterversion -o jsonpath='{.spec.clusterID}'
```
- Most reliable identifier
- Globally unique per cluster
- Cannot be spoofed or confused

#### Secondary Check: Server URL
```bash
oc whoami --show-server
```
- Unique per cluster (confirmed by user)
- Fast check, no API calls needed
- Catches cluster context switches

#### Tertiary Check: Infrastructure Name
```bash
oc get infrastructure cluster -o jsonpath='{.status.infrastructureName}'
```
- Often contains cluster ID
- Additional safety check
- Can prompt for confirmation if mismatch

### 4. Error Handling

If verification fails, the script:

1. **Displays clear error** with expected vs actual cluster info
2. **Shows exact login command** with correct cluster ID
3. **Exits immediately** to prevent wrong-cluster operations
4. **Provides context** about where in the process the check failed

Example error output:
```
❌ CLUSTER MISMATCH ERROR!

Expected cluster:
  ID: YOUR_CLUSTER_NAME
  UUID: abc123-def456-...
  Server: https://api.cluster1.example.com:6443

Current cluster:
  UUID: xyz789-ghi012-...
  Server: https://api.hive-cluster.example.com:6443
  Infrastructure: hive-cluster-abc123

You are connected to the WRONG cluster!

Context: After Hive pause - returning to target cluster

Please login to the correct test cluster:
  ocm backplane login YOUR_CLUSTER_NAME
```

## Usage

### First Time Setup (Phase 1)

1. **Build images** (can be done without cluster access)
2. **Optional**: If logged into test cluster, provide cluster ID early

Phase 1 will save cluster info to config for all subsequent phases.

### Main Workflow (Phase 3)

1. Script prompts for cluster ID (e.g., `YOUR_CLUSTER_NAME`)
2. Verifies you're logged into a cluster
3. Saves cluster UUID, server URL, and ID to config
4. All verification points after this check against these saved values

### Critical Verification Points

#### After Hive Sync Pause (Phase 3)
```
⚠️  CRITICAL: Verify you are connected to the TARGET test cluster
             NOT the Hive management cluster!
```

The script re-verifies cluster connection after the pause for Hive operations to catch accidental context switches.

#### After Hive Sync Restoration (Phase 8)
Same critical re-verification happens after restoring Hive sync.

## Shared Verification Function

All scripts source `cluster-verification.sh` which provides:

### `verify_cluster(context)`
Verifies current cluster connection matches saved config.

Parameters:
- `context` - Description of where verification is happening (for error messages)

### `save_cluster_info(cluster_id, config_file)`
Saves cluster information to config file.

Parameters:
- `cluster_id` - User-provided cluster identifier
- `config_file` - Path to config file (default: `.camo-pko-test-config`)

### `confirm_operation(operation_type, commands...)`
Shows cluster context and commands to be executed, waits for explicit confirmation.

Parameters:
- `operation_type` - Description of the operation (e.g., "DELETE OLM RESOURCES")
- `commands` - Array of oc commands that will be executed

Example output:

## Operation Confirmation

### Double-Check Before Destructive Operations

In addition to cluster verification at the start of each phase, the scripts now require explicit confirmation before any destructive operation (delete or apply).

### When Confirmation is Required

Confirmation prompts appear before:

#### Phase 4 (Remove OLM)
- Scaling down deployment to 0 replicas
- Deleting Subscription, CSV, and CatalogSource

#### Phase 5 (Deploy PKO)
- Applying ClusterPackage manifest

#### Phase 8 (Cleanup)
- Deleting ClusterPackage
- Manually deleting deployment (if cleanup incomplete)

### What You See

Before each operation, you'll see:
1. **Operation type** - What's about to happen
2. **Target cluster** - Name, ID, and server URL
3. **Exact commands** - The actual `oc` commands to be executed
4. **Confirmation prompt** - Requires typing "yes" (not just "y")

### Example Workflow

1. Script reaches a delete/apply operation
2. Shows confirmation dialog with cluster context
3. Lists exact commands that will run
4. Waits for "yes" confirmation
5. Executes commands if confirmed, exits if cancelled

### Safety Benefits

- **Catch context switches** - Even if you're on the wrong cluster, you see the cluster name before destructive operations
- **Review commands** - See exactly what will execute before it happens
- **Explicit confirmation** - Must type "yes" (not just "y") to proceed
- **Easy abort** - Any response except "yes" cancels the operation

## Benefits

1. **Prevents wrong-cluster operations** - Multiple verification layers
2. **Operation confirmation** - Explicit approval before each destructive action
3. **Clear error messages** - Shows exactly what's wrong and how to fix it
4. **Context-aware** - Knows where in the process verification failed
5. **Automatic** - No manual checks needed, built into every phase
6. **Fast** - Server URL check is instant, UUID check is one API call
7. **Visible commands** - See what will execute before it runs

## Recovery

If you accidentally run on the wrong cluster:

1. **Stop immediately** if you catch it
2. **Check backup directory** - Phase 3 creates backups before any changes
3. **Restore from backup** if needed (see main README)
4. **Login to correct cluster** using the cluster ID from error message
5. **Resume from current phase** - verification will pass once on correct cluster

## Testing

To test the verification:

1. Run phase 3 on correct cluster
2. Login to a different cluster: `ocm backplane login other-cluster`
3. Try running phase 4
4. Should see cluster mismatch error and exit immediately
5. Login back to correct cluster: `ocm backplane login YOUR_CLUSTER_NAME`
6. Phase 4 should now pass verification
