# PKO Testing Gaps Identified (2026-05-08)

## Critical Gap: OLM-to-PKO Migration Path Not Tested

### What happened

During CAMO's stage-01 OLM-to-PKO migration, PKO refused to adopt the
pre-existing `ClusterRoleBinding/configure-alertmanager-operator-prom`
with the error:

```
refusing adoption, object rbac.authorization.k8s.io/v1,
Kind=ClusterRoleBinding /configure-alertmanager-operator-prom
not owned by previous revision
```

This was caused by missing `collision-protection: IfNoController`
annotations on RBAC resources. The issue was not caught in any of our
16-scenario test matrix because **none of the tests simulated an actual
OLM deployment before migrating to PKO**.

### Why it wasn't caught

All test scenarios either:
1. Deployed ClusterPackage on clean clusters with no pre-existing resources
2. Tested on integration clusters that had already migrated (old cleanup
   Job deleted the CRB-prom before our testing)
3. Tested PKO-to-PKO upgrades without OLM history

The test cluster was never in a state that simulated what a real OLM
cluster looks like before migration.

### What OLM leaves on a managed cluster

When CAMO is deployed via OLM (SelectorSyncSet), these resources exist:

**Synced by OLM SelectorSyncSet (from hive):**
- CatalogSource (configure-alertmanager-operator-registry)
- Subscription (configure-alertmanager-operator)
- ClusterRoleBinding (configure-alertmanager-operator-prom)

**Created by OLM from CSV:**
- Deployment (configure-alertmanager-operator) — ownerRef to CSV,
  controller: false
- ServiceAccount (configure-alertmanager-operator) — no ownerRef

**NOT created by OLM (only exist under PKO):**
- Role (configure-alertmanager-operator)
- RoleBinding (configure-alertmanager-operator)
- ClusterRole-view (configure-alertmanager-operator-view)
- ClusterRole-edit (configure-alertmanager-operator-edit)
- ClusterRoleBinding-view
- ClusterRoleBinding-edit

## Required Test Scenarios to Add

### OLM Simulation Tests

Before testing PKO migration, the test must create all OLM resources
that would exist on a real cluster:

```bash
# Simulate OLM SelectorSyncSet resources
oc apply -f olm-simulation/catalogsource.yaml
oc apply -f olm-simulation/subscription.yaml
oc apply -f olm-simulation/clusterrolebinding-prom.yaml

# Simulate OLM CSV-created resources
oc apply -f olm-simulation/serviceaccount.yaml
oc apply -f olm-simulation/deployment.yaml
```

Then deploy ClusterPackage and verify PKO can adopt all pre-existing
resources.

### Test Matrix Additions

| Test | Description | Validates |
|------|-------------|-----------|
| OLM-1 | Create all OLM resources, then deploy PKO ClusterPackage | collision-protection adoption works |
| OLM-2 | OLM resources with ownerReferences (CSV controller:false), then PKO | IfNoController handles non-controller owners |
| OLM-3 | OLM resources with hive labels, then PKO | Labels don't interfere with adoption |
| OLM-4 | OLM resources + active OLM SSS (simulated), then PKO | Cleanup Job deletes OLM artifacts |
| OLM-5 | Partial OLM state (some resources missing), then PKO | PKO creates missing, adopts existing |
| OLM-6 | OLM resources + PKO deploy + verify operator health | End-to-end functional validation |
| OLM-7 | OLM resources + PKO deploy + PKO upgrade | Migration then upgrade works |

### Collision Protection Verification

For each resource in deploy_pko/, verify:
1. Resource has `collision-protection: IfNoController` annotation
2. If the resource pre-exists from OLM, PKO adopts it
3. If the resource doesn't pre-exist, PKO creates it
4. After adoption, the resource has PKO ownerReference

### Per-Operator OLM Resource Mapping

Each operator needs its own OLM simulation fixtures based on what
OLM/SSS actually deploys. This was documented for CAMO via the
`investigate/check-olm-ownership.sh` script run on a live stage cluster.

Other operators need similar investigation before testing.

## Existing Test Gaps (from our 16-test matrix)

### Tests that passed but were insufficient

| Original Test | Gap | Required Addition |
|---------------|-----|-------------------|
| Test 1 (fresh deploy) | No pre-existing resources | Add OLM simulation before deploy |
| Test 7 (static→versioned) | Pre-PR image, not OLM | Need real OLM resources |
| Tests 2-4 (PKO upgrades) | Valid but missing OLM prerequisite | Run after OLM migration test |

### Infrastructure gaps

- No automated way to create OLM simulation state
- No verification that collision-protection is set on all resources
- No per-resource adoption verification after migration
- No comparison of OLM-deployed resources vs deploy_pko/ resources
